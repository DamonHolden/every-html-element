This is a section element

Generally speaking, a section represents a generic standalone section of the document. The key word here is generic, as the section element should only be used if there is not a more specific element to represent it.

For example, a list of search of search results does not have a specific element, so this could be wrapped in a section element.

As with other segmentation elements, a section element should be identified by including a heading element. However there may be situations where this wouldn't make sense.

This is an article element

While not visually special, article elements should be used to wrap content that can be reused, like user comments, product cards, or a widget.

Each article element should include a heading element as its first child.

If reasonable, articles can also be nested.

This is an aside element

While visually indistinguishable in most cases, an aside element represents content that is only indirectly related to the documents main content.

Asides are commonly styled as sidebars or call-out boxes to distinguish themselves from other content in the document.

The blockquote element

The following text is contained within a blockquote element. I'll let Mozilla do the explaining for this one.

The blockquote HTML element indicates that the enclosed text is an extended quotation. Usually, this is rendered visually by indentation (see Notes for how to change it). A URL for the source of the quotation may be given using the cite attribute, while a text representation of the source can be given using the cite element.

By the way, that cite attribute isn't purposed to provide any visual content to the end-user. Instead, th URL value of the cite attribute is intended to be used by scripts as metadata for some other purpose, though conceptually the URL provided by the cite attribute could be used by the webpage to render something on the screen.

The div element

The prior h1 tag and this text is all contained within a div element. This element is effectively as generic as an element can get, as it doesn't represent anything specific other than simply a divider for page content.

While a div element could be (and is commonly) seen as a flexible element, as it could be used as a segmentation element in almost any scenario, it should really only be used when no other element is appropriate. This is noted in the MDN documentation as well as on the WHATWG's HTML spec:

Authors are strongly encouraged to view the div element as an element of last resort, for when no other element is suitable. Use of more appropriate elements instead of the div element leads to better accessibility for readers and easier maintainability for authors.

A div's representation of a purely generic element also means that there is no styling that is applied to the element by default, other than that in most browsers, a div will be new-lined by default.

The dl, dt, and dd elements

element definitions:

dl
represents a description list, enclosing a list of groups of terms and descriptions. Common use cases are glossaries or to display metadata.
dt
element used when specifying a term in a description/definition list.
dd
element that'a contents provide the description, definition, or value for the preceding term.

While the dl, dt, and dd elements are fairly straightforward to understand, terms and descriptions do not have to be one-to-one pairs. It's possible to define a single dt with multiple dd's, multiple dt's with a single dd, or even multiple dt's with multiple dd's.

the figure and figcaption elements

the following picture and caption is enclosed within a figure element. The caption below the picture, is contained within a figcaption element, if you can believe that.

The figure element itself represents self-contained content, with an optional caption (implemented using figcaption). Typically the actual content of the figure will be some kind of image, diagram, code-snippet, etc.

The hr element

The hr element is thematic-break between elements. This could be a shift of topic in a section, for example.

As far a specification goes, this element should really be used to section paragraph level elements. While the hr element may, by default, visually render a horizontal line and spacing, it should be used logically. A demonstration of using this element in technically the wrong way is within this very document. each section of this page is separated using the hr element, mainly for its visual rendering, but this is really going against HTML spec, as its not being used as a thematic-break between paragraph-level elements.

While the usage of hr in this document suffices for the purpose of its usage here, as far as best-practice is concerned, you should aim to achieve the spacing and horizontal line of the hr element with CSS, as opposed to using the hr element itself to achieve this.

The li, ol, ul, and menu elements

The li element

The li element represents an item in a list and must be the child of one of the following three elements:

  • an ol element
  • a ul element
  • a menu element

The ol element

An ol element should be used for a list if the order is meaningful, like steps in a recipe. For example:

  1. Pick up phone.
  2. Call takeaway.
  3. Order a pizza.
  4. Wait for delivery.
  5. Eat pizza.

The ul element

a ul element should be used for a list if the order is meaningless, like a list of individuals. For example:

  • Boba Fett
  • Jin Darin
  • Grogu
  • General Grievous
  • Darth Vader

The menu element

similarly to the ul element, the menu element represents a list of unordered items. The key difference between the two is while a ul is intended for only displaying those items, a menu is used when those items also include user interaction. For example, a toolbar:

    • The <p> element

    The <p> element represents a paragraph. while you may typically think this would be limited to a block of text, in HTML, a paragraph can be any structural grouping of related content. This is important to remember.

    Almost all text groups in this document are p elements, which by default provides spacing above and below those blocks.

    While an empty p element will render empty space, you shouldn't use a p element in this way, not only because it goes against standards, but also for accessibility concerns. For example, if a reader uses screen-reading technology, that technology may announce an empty p element, but there will be no content to read, resulting in a confusing experience.

    The pre element

    The p element represents preformatted text, which is to be presented exactly as written in the HTML file it is sourced from, including any white space. Typically this element is rendered monospace font by default.

    The tricky aspect of implementing a pre element is character escaping. Symbols desired to be placed inside a pre element that are interpreted as HTML syntax will need to be escaped. For example, the '<' symbol will need to be replaced with '&#60;' for that particular symbol to be interpreted as text. This consideration is not exclusive to the pre element, but is most commonly ran into when writing one.

    A useful web-tool for looking-up symbols and their Html escapes can be found here.

    While pre elements are an effective way to display text-based images, its important for accessibility to ensure that the pre element is considered an image by the browser. for example a plain pre element with an ASCII picture may be read by a screen-reader as actual text, which would be a pretty obnoxious experience.

    The a element

    The a element, along with the href attribute, creates a hyperlink to anything that URL can address.

    The contents of an a element should clearly indicate the links destination. As a side-note, the link created by an a tag can also be actuated by hovering over the link and pressing the enter key.

    Here are some a element examples:

    • This a element will download this page as a .html file.
    • This a element will go to MDN's page on the a element. (opens in new tab)
    • This a element will move take you to the top of this page.
    • This a element is a link to my telephone number.
    • This a element is a link to my email address.

    There is a variety of way an a element can be utilized due to the variety of attribute the element has access to, so its important to consider these attributes when using an a element.

    The usage of an a element can also have consequences for the security of both the site and it's users. This is a pretty important topic whenever brought up in web development, so I will simply provide a link to MDN's page on the Referer header (opens in new tab) if you would like to learn more about this.

    a elements should also be absolutely clear as to what they will direct the user to. The rule of thumb is that the link should make sense completely out of context. While it may be elegant to have a link that fits nicely into the sentence of a paragraph, its more important to make sure the user knows exactly where they are being directed to.

    Additionally to the previous point, there should also be some indication as to what action will happen when a link is followed. For example, if a link opens in a new tab, its best to include that information in the link text.

    the a element and its usage is a relatively deep topic, so if you want to know more information, it's a good idea to do some further research, as this is a very commonly used element.

    The abbr element

    the abbr element represents an abbreviation or acronym. The optional attribute can then be used to provide a description for the abbreviation.

    an abbr element, when hovered over with the mouse cursor, is presented by browsers as a tooltip.

    The b element

    The be element is used to draw the readers attention to content which does not otherwise have special importance

    Identifying an appropriate use-case for the b element can be tricky, as there may be another element that is better purpose in many cases. A good rule of thumb for me is to use it for words and phrases that are effectively especially contextual. For example, keywords in a summary, product names in a review, or ingredients in a recipe.

    While the b element may have historically been used to make text boldface. Styling information in this way has been deprecated, so use CSS styling instead to achieve this.

    The bdi element

    The bdi element tells the browser's bidirectional algorithm to treat the elements's text contents in isolation from it's surrounding text. Browsers will usually handle bidirectional text pretty well, but in situations when you notice this going wrong, thats where this element comes in. For example:

    • اَلأَعْشَى - 1st place
    • Jerry Cruncher - 2nd place

    The bdo element

    The bdo element works in a similar nature to the bdi element, in that it is purposed to handle text direction. More specifically, it is purposed to override the current text direction. For example:

    • This text will go left to right.
    • This text will go right to left.

    It's worth pointing out that while the bdo element will override text direction, It will not affect individual character orientation.

    The br element

    The br element is used for creating line-breaks in text, where the division of lines is significant. Common examples are addresses or poems.

    A couple of noteworthy points here. A br element should not be used purely to add visual spacing or anything of the such, and should be used for its exact purpose. Also, using br elements for visual purposes is problematic for assistive reading technology, where a br element may be announced, but doesn't make sense to be.

    The cite element

    the cite element is used to cite creative work by including the title of that work. This may be used for films, books, songs, etc.

    one important thing to note here is that the W3C and WHATWG spec say two different things about the cite element. While the W3C state that the contents of a cite element may include the name of the work's author, WHATWG state that the name should never be included. It's a developers choice really, but if you want to write markup to an exact specification, I would always go with whatever the WHATWG spec states.

    Also, browsers will typically style cite elements in italic by default.

    the <code> element

    the <code> element should be used to indicate the text contents as computer code.

    Typically by default, the <code> element will render contained text with monospace font.

    To represent multiple lines of code, its generally best to wrap the <code> element within a <pre> element.

    The <data> element

    the <data> element links a piece of content with a machine-readable translation using the value attribute. For example, product names with a product number:

    • small duck
    • large duck
    • biiiiiiiiiiiiig duck

    As you can probably see, in the above list, there is no visual output of the data element by default, its purpose is really to allow human-readable content to also have a machine-readable value. This could be useful is you wanted to manipulate content in a script for example.

    In the prior list the data element can be observed upon page inspection.

    The <dfn> element

    The <dfn> element is used to indicate a term being defined within it's context.

    The <dfn> element is useful in couple of main ways.

    1. It can be used to visually make term definitions clear to a reader.
    2. It can also be linked to throughout the rest of the document so a reader always has convenient access to the definition of a term.

    The <em> element

    The <em> element marks text with emphasis.

    while the contents of a <em> element will typically render as italicised text in most browsers by default, the element should not be used to solely achieve this. Instead, use CSS to achieve this.

    The <i> element

    The <i> element represents text that is set of from the normal text.

    historically, the <i> element has been used as a element to purely italicise text, but developers should use it for its current semantic purpose.

    The <kbd> element.

    The <kbd> element is used to represent text denoting user keyboard , or any other text entry device.

    By default, most browsers will render <kbd> text in monospace, but this can be changed with CSS.

    The <mark> element

    The <mark> element represents text which is marked for reference purposes, due to its relevance in another context.

    While the <mark> element is typically used to mark the the relevancy of text of a document in new context, it can also be used to mark parts of a document relating to a search result.

    As is the case with many other elements, do not use the <mark> purely for its styling purposes. While the element does, by default in most browsers, render the marked text as if it were highlighted with a yellow highlighter, if you wish to purely achieve this visual output, use CSS.

    The <q> element

    The <q> element is used to indicate enclosed text as an inline quote.

    Ensure to use the <q> element for inline quotes only. For Long quotes, use the <blockquote> element.

    By default, most browsers will render quote marks round the contents of a <q> element.

    The <ruby>, <rp>, and <rt> elements

    The <ruby> element

    the <ruby> element represents annotations rendered with base text.

    The <ruby> element is commonly used for showing the pronunciation of east east-asian characters. However, you could conceptually use it to annotate other kinds of text.

    The <rt> element

    The <rt> element the annotation of a <ruby> element.

    As a component element of the <ruby>, any instance of an <rt> element must be contained inside a <ruby> element.

    The <rp> element

    The <rp> element is used to provide fallback parentheses for browsers that do not support the display of ruby annotations using the <ruby> element.

    Typically when being used, the two of these elements would be used for opening and closing parentheses, but alternatives are allowed.

    The <s> element

    The <s> element is used to represent things that are no longer relevant/accurate.

    While a similar use-case, the <s> element shouldn't be used to indicate document edits. The <del> and <ins> elements are purposed for this.

    Typically, browsers will render a strikethrough line through the contents of a <s> element.

    The <samp> element

    The <samp> element is used to enclose inline text which represents sample output from a computer program.

    The <small> element

    The <small> element is used to represent side-comments and small print.

    While HTML standards are fairly loose on how the <small> element can be used, you should still aim to use it semantically, and not purely for it's inherent styling.

    The <span> element

    The <span> element is a generic inline container for content, which semantically does not inherently represent anything.

    Being generic, the <span> element should be used only be used when no other semantic element is better suited. In this way, <span> is very similar in purpose to the <div> element, the difference being <span> is an inline element and <div> is a block-level element.

    Typically, the <span> element is used when groups of elements share the same styling or attributes.

    The <strong> element

    The <strong> element indicates that it's contents have strong importance.

    while the <strong> element renders typically renders it's contents in boldface, but this should not be the sole reason to justify the element's use. Remember to use <strong> semantically.

    MDN's documentation on the <strong> element includes some really good commentary on how the <strong> element fits in with other similar semantic elements. Its important to remember that no two elements in HTML fulfil the same purpose, they all have a purpose.

    The <sub> element

    The <sub> element specifies text which should be displayed as subscript.

    The <sub> element should only be used to change the position of text with a typographical convention or standard to comply to. In other words, if you just want the styling of this element, use CSS instead.

    Appropriate uses for the <sub> element include:

    • Footnote numbers (According to the computations by Nakamura, Johnson, and Mason1 this... )
    • Atom numbers in chemical formulas (H2O)
    • Mathematical variable numbers (x1 ... xn)

    The <sup> element

    The <sup> element specifies text which should be displayed as superscript.

    The <sup> element should only be used to change the position of text with a typographical convention or standard to comply to. In other words, if you just want the styling of this element, use CSS instead.

    Appropriate uses for the <sup> element include:

    • Exponents (x3)
    • Superior lettering (Mlle)
    • Ordinal numbers (3rd)

    The <time> element

    The <time> element represents a specific period in time.

    The datetime attribute can also be used with the <time> element to provide a machine-readable format for the date/time content of the <time> element. This could then be used for some other functionality, like search results or reminders.

    Alternatively, omitting the datetime attribute from the <time> element will result in the contents of the element implicitly filling the attributes value. So if using the element in this way, ensure that the contents are machine rereadable.

    The <u> element

    The <u> element represent text that should be rendered in a way that indicates a non-textual annotation.

    Only use the element is the annotation is semantically meaningful. If you are using it purely fo styling purposes, then use the <span> element paired with CSS instead.

    In most cases, you'll probably find that you shouldn't be using the <u> element. The use-case of using it to annotate misspelled text is valid because there is semantic meaning to the annotation. If you cannot give a similar reason for why you are using the <u> element in your markup, then you probably shouldn't be using it.

    The <var> element.

    the <var> element represents the name of a variable in a mathematical or programming context.

    Similarly with other inline text semantics, if the <var> element is being used for its styling output, then it is being used wrong, and in situations like that, the <span> element should be used.

    The <wbr> element.

    the <wbr> element represents a position in text where the browser may optionally break a line.

    a brilliant example of the <wbr> element would be in a URL, where you may use it avoid leaving a punctuation mark at the end of a line, which may be mistaken by a human reader to be the end of the URL.

    The <img> element.

    The <img> element simply embeds an image in the document.

    The src attribute of an <img> element contains the path to the image, and is a required attribute of the element.

    The alt attribute of the <img> element provides a text description of the image. While not mandatory, it is really useful to utilise this attribute as not only will it render something if the image fails to load, it is also typically read by screen reading technology.

    One of the more interesting things I found when researching about the <img> element were the supported image formats. From reading The HTML standards, there doesn't appear to be much commentary on what best practices are around image formats. However, the MDN documentation includes a very well-put-together Image and file type guide (opens in new tab).

    Aside from what is mentioned in this section about the <img> element, there is plenty more information on the element itself and more-generally the subject of the utilisation of images on the web. It is definitely worth looking into as you will most likely be working with images in your web-projects a lot.

    The <video> element.

    The <video> element embeds a media player into the document.

    The <video> element can also be used to play audio, but the <audio> element is better suited for this.

    Just like the <img> element, their is a lot of documentation surrounding the <video> element and its usage. I'll leave it at the basic-level here, but a good place to start diving deeper is MDN's video and audio content tutorial.

    The <audio> element

    The <audio> element is used to embed sound content in a document.

    Again, media elements can be complex in their functionality and behavior. While this document provides a basic overview, MDN's video and audio content tutorial is a good place to start delving deeper into the <audio> element.

    The <track> element

    The <track> element is used as a child of the <video> and <audio> elements. It allows you to specify time based data for the parent media element.

    The time based data used in a <track> element is formatted in WebVTT.

    The <track> element can be used to provide subtitles, captions, descriptions, chapters, and other metadata to the targeted media.

    The <map> and <area> elements

    the <map> element is used to define an image map (an image with clickable link areas).

    The <area> element defines a clickable area inside an image map.

    A <map> element must include the name attribute, which provides a reference to the <map> element which an <img> element can use to apply the map to itself.

    The shape and coords attributes are used in an <area> element to specify the dimensions of a clickable-area within an image-map
